# 版本记录：v1.0-nea-runway-distance

## 1. 本轮目标
本轮目标是为当前目录中的 `.nea` 文件建立一套可复用的 Python 计算流程，直接输出跑道起终点距离。

这次优先处理的原因是：原始数据已经齐备，但仓库里没有可直接复用的解析脚本，人工逐个查看和手算不可持续。

本轮任务类型：功能补充 + 基线工具搭建。

## 2. 背景与上下文
当前仓库中只有多份 `.nea` 原始轨迹文件，没有现成的 Python 程序，也没有可参考的正式版本记录。

我在开始前检查了 `custom/notes/`，未发现相关历史记录，因此本轮作为首轮方案建立基线。

已确认这些 `.nea` 是 NMEA 文本日志，包含 `GPRMC / BDRMC / GPGGA / BDGGA` 等定位句子，适合直接提取有效经纬度后计算首尾点直线距离。

## 3. 问题分析
表面问题是：用户需要三条跑道在三种测量方式下的长度结果。

核心矛盾是：文件格式是原始定位日志，不是已经整理好的表格；如果不先建立统一解析口径，就无法稳定批量计算。

本轮已经把需求收敛为“直接计算每个文件中首个有效定位点到最后一个有效定位点的距离”，不再做轨迹累加。

## 4. 候选方案比较
方案一：轨迹累加。
- 核心思路：把相邻定位点逐段相加，得到行进路径长度。
- 优点：适合分析轨迹总长度。
- 缺点：和当前“起终点距离”的需求不一致，且更容易被抖动和绕行放大。
- 结论：放弃。

方案二：首尾点直线距离。
- 核心思路：提取每个文件的首个有效点和最后一个有效点，用大地距离计算二者间距离。
- 优点：和当前需求直接对应，实现简单，结果稳定，可批量输出。
- 缺点：依赖首尾记录本身足够代表起止点；如果记录前后有长时间空转或误采样，结果会受影响。
- 结论：采用。

## 5. 最终决策
最终采用“首尾有效定位点直线距离”作为跑道长度口径。

关键决策是：
- 兼容 GP 和 BD 两套报文
- 只接受有效定位状态的记录
- 按文件名自动识别跑道名、测量方式和重复编号
- 输出明细 CSV，方便后续人工复核和二次统计

## 6. 具体实现
新增了两个文件：

- `custom/nea_length/nea_tools.py`
- `custom/nea_length/calculate_runway_distance.py`

其中：
- `nea_tools.py` 负责解析 NMEA 坐标、抽取有效定位点、解析文件名分组、计算首尾距离
- `calculate_runway_distance.py` 负责批量读取当前目录下的 `.nea` 文件、生成结果、导出 CSV、打印汇总

## 7. 本轮优点
当前方案的优点是：

- 解析逻辑和批处理逻辑分离，后续扩展更容易
- 对不同命名风格的文件名做了兼容
- 计算口径明确，结果便于复现
- 可直接处理当前目录全部 `.nea` 文件

## 8. 本轮缺点与代价
当前方案的已知缺点是：

- 结果完全依赖文件首尾记录是否能代表真正起点和终点
- 没有进一步做异常跳点筛除或轨迹平滑
- 不同测量方式的结果差异较大时，脚本本身不会判断原因，只负责计算

这属于“先跑通、后优化”的版本，当前接受这个妥协是为了先把计算链路建立起来。

## 9. 验证与结果
已完成验证：

- 运行 `python custom/nea_length/calculate_runway_distance.py --input-dir .`
- 成功读取当前目录 18 个 `.nea` 文件
- 成功生成明细 CSV：`custom/nea_length/runway_distance_results.csv`

验证结果表明脚本可以正常工作，且三条跑道按三种方式都能得到对应首尾距离。

## 10. 本轮结论
本轮确认：当前 `.nea` 文件可以稳定解析，且用首尾点直线距离就能得到用户要求的跑道长度结果。

本轮也确认：后续若要做更严格的实验分析，可以在现有模块上继续加“异常点过滤”“按时间截取有效区间”“结果可视化”等能力。
